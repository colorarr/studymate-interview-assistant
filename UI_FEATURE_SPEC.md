# StudyMate Interview Assistant UI 功能说明文档

> 配套设计系统见：[DESIGN_SYSTEM.md](DESIGN_SYSTEM.md)。该文档补充了产品气质判断、当前视觉语言提炼、完整 Design System 规范与可交付给前端的 design tokens。

## 1. 文档目的

本文档用于交付给 UI 设计，帮助设计师明确：

- 项目是什么
- 用户是谁
- 包含哪些页面
- 每个页面需要承载什么功能
- 页面之间如何流转
- 重点状态和交互有哪些

本文档聚焦于页面设计与交互表达，不涉及后端实现细节。

## 2. 项目概述

### 2.1 项目名称

`StudyMate Interview Assistant`

### 2.2 项目定位

这是一个面向面试八股场景的 AI 知识库问答系统。

用户可以上传自己的面试资料，例如：

- PDF
- Word
- Markdown
- TXT

系统会对资料进行解析、切片、向量化，然后用户可以围绕资料内容进行 AI 问答。

### 2.3 核心价值

- 帮助用户整理和管理面试资料
- 将静态资料转成可交互的知识库
- 通过 AI 问答提升复习效率
- 支持引用来源展示，增强答案可信度

## 3. 目标用户

目标用户主要是：

- 准备 Java 后端面试的学生
- 社招求职中的后端开发者
- 需要整理技术八股资料的学习者

用户特征：

- 高频使用手机，也会使用 Web
- 偏重实用和效率
- 希望快速找到知识点并进行追问
- 对“文件管理 + AI 问答”组合场景敏感

## 4. 设计方向建议

整体风格建议：

- 专业
- 清晰
- 高效率
- 有学习工具感

建议关键词：

- 学习助手
- 知识库
- 面试准备
- 智能问答

视觉重点建议：

- 知识库列表要清晰
- 文件管理要直观
- 聊天问答页要作为核心页面重点设计
- 回答中的“引用来源”需要有明显但不抢主内容的展示方式

## 5. 产品结构

第一版建议包含以下页面：

- 登录页
- 注册页
- 首页/知识库列表页
- 创建知识库弹层或页面
- 知识库详情页
- 文件上传页或上传弹层
- 聊天问答页
- 历史会话页
- 我的页面

## 6. 页面流转关系

主流程如下：

1. 用户进入系统
2. 登录或注册
3. 进入知识库列表页
4. 创建知识库
5. 进入知识库详情
6. 上传资料文件
7. 文件处理完成后进入聊天页
8. 基于该知识库进行问答
9. 查看历史会话并继续追问

## 7. 页面功能说明

### 7.1 登录页

页面目标：

- 让用户快速进入系统

页面功能：

- 输入用户名
- 输入密码
- 登录按钮
- 跳转注册入口

页面元素建议：

- Logo / 项目名
- 简短项目介绍
- 输入框
- 主按钮
- 次级跳转按钮

设计重点：

- 登录入口清晰
- 移动端单手操作方便
- 表单不要复杂

状态说明：

- 默认状态
- 输入中状态
- 登录中状态
- 登录失败提示状态

### 7.2 注册页

页面目标：

- 完成新用户注册

页面功能：

- 输入用户名
- 输入昵称
- 输入密码
- 确认注册
- 返回登录

设计重点：

- 表单结构简洁
- 注册成功反馈明确

状态说明：

- 默认状态
- 校验失败状态
- 注册成功状态

### 7.3 首页 / 知识库列表页

页面目标：

- 展示当前用户已有知识库
- 作为进入核心功能的主入口

页面功能：

- 查看知识库列表
- 搜索知识库
- 创建知识库
- 进入知识库详情
- 删除知识库

每个知识库卡片建议展示：

- 知识库名称
- 简介描述
- 文件数量
- 分片数量
- 状态
- 更新时间

设计重点：

- 列表信息层级清楚
- 创建入口明显
- 空状态要友好

状态说明：

- 空知识库状态
- 正常列表状态
- 知识库处理中状态
- 知识库处理失败状态

### 7.4 创建知识库页 / 弹层

页面目标：

- 让用户快速创建一个新的知识库

页面功能：

- 输入知识库名称
- 输入知识库描述
- 确认创建

设计重点：

- 轻量
- 易理解
- 不打断主流程

建议：

- Web 可优先用弹层
- 移动端可优先用独立页或底部弹出层

### 7.5 知识库详情页

页面目标：

- 查看某个知识库的完整信息
- 管理该知识库下的文件
- 进入聊天问答

页面功能：

- 查看知识库基础信息
- 查看文件列表
- 上传文件
- 删除文件
- 重新解析文件
- 进入问答页

页面结构建议：

- 顶部：知识库名称、描述、状态
- 中部：文件列表
- 底部或悬浮按钮：上传文件、开始问答

文件列表项建议展示：

- 文件名
- 文件类型
- 文件大小
- 上传时间
- 解析状态
- 操作按钮

设计重点：

- 文件状态必须清楚
- 问答入口要明显
- 上传入口要方便

状态说明：

- 无文件状态
- 文件处理中状态
- 文件解析成功状态
- 文件解析失败状态

### 7.6 文件上传页 / 上传弹层

页面目标：

- 完成文件上传

页面功能：

- 选择文件
- 展示待上传文件列表
- 发起上传
- 展示上传进度
- 展示处理结果

支持类型：

- PDF
- Word
- Markdown
- TXT

设计重点：

- 上传反馈要清晰
- 支持失败提示
- 成功后能快速返回详情页

状态说明：

- 待上传状态
- 上传中状态
- 上传成功状态
- 上传失败状态

### 7.7 聊天问答页

页面目标：

- 作为整个产品的核心使用页面
- 用户围绕知识库进行问答和追问

页面功能：

- 查看当前知识库名称
- 查看当前会话标题
- 输入问题
- 发送问题
- 流式接收 AI 回答
- 查看引用来源
- 查看历史消息

页面结构建议：

- 顶部：返回、知识库名称、会话信息
- 中部：聊天消息区
- 底部：输入框、发送按钮

消息区包含：

- 用户消息气泡
- AI 消息气泡
- AI 正在生成中的流式状态
- 引用来源区域

引用来源建议展示：

- 文件名
- 摘要片段
- 可折叠查看

设计重点：

- 聊天体验要流畅
- AI 流式回答过程要有“正在生成”的感知
- 引用来源不要影响主阅读
- 移动端输入体验要顺手

关键状态：

- 空会话状态
- AI 正在回答状态
- 回答完成状态
- 回答失败状态
- 网络异常状态

### 7.8 历史会话页

页面目标：

- 管理用户的历史聊天记录

页面功能：

- 查看会话列表
- 点击进入旧会话
- 删除某个会话

每个会话项建议展示：

- 会话标题
- 所属知识库
- 最后一条消息摘要
- 更新时间

设计重点：

- 列表便于快速定位
- 历史记录和当前知识库关系要清楚

### 7.9 我的页面

页面目标：

- 展示用户个人信息和常用操作入口

页面功能：

- 查看用户昵称
- 查看用户名
- 退出登录

第一版可以保持轻量。

## 8. 核心交互说明

### 8.1 创建知识库

交互过程：

1. 点击创建知识库
2. 输入名称和描述
3. 创建成功后进入知识库详情或返回列表

### 8.2 上传资料

交互过程：

1. 用户进入知识库详情
2. 点击上传文件
3. 选择一个或多个文件
4. 上传并进入处理状态
5. 处理完成后文件状态变为可用

### 8.3 开始问答

交互过程：

1. 用户进入聊天页
2. 输入问题
3. 提交后消息立即进入聊天区
4. AI 通过 SSE 流式输出
5. 最终展示完整回答和引用来源

### 8.4 查看历史会话

交互过程：

1. 用户进入历史会话页
2. 点击某条会话
3. 回到对应聊天内容
4. 支持继续追问

## 9. 页面状态清单

UI 设计时建议重点补齐以下状态页或状态样式：

- 空数据状态
- 加载中状态
- 上传中状态
- 处理中状态
- 成功状态
- 失败状态
- 删除确认状态
- 网络异常状态
- AI 流式生成中状态

## 10. 组件需求建议

建议优先沉淀这些可复用组件：

- 知识库卡片
- 文件列表项
- 状态标签
- 聊天气泡
- 引用来源卡片
- 空状态组件
- 上传进度组件
- 底部输入框组件
- 删除确认弹层

## 11. 重点设计页面排序

如果 UI 设计需要排优先级，建议按这个顺序出稿：

1. 聊天问答页
2. 知识库列表页
3. 知识库详情页
4. 文件上传页
5. 登录/注册页
6. 历史会话页
7. 我的页面

原因：

- 聊天问答页是核心页面
- 知识库页和详情页承载主要业务入口
- 上传页直接影响主流程体验

## 12. 设计交付建议

建议 UI 最终至少输出以下内容：

- Web 端页面设计稿
- 移动端页面设计稿
- 核心组件样式规范
- 颜色 / 字体 / 间距规范
- 交互状态稿

重点要覆盖：

- 登录状态
- 空状态
- 处理中状态
- AI 回答流式状态
- 引用来源展开状态

## 13. 一句话总结

这是一个“围绕面试资料构建知识库，并通过 AI 进行流式问答”的学习工具型产品。

UI 设计重点应放在：

- 知识库管理清晰
- 上传流程顺畅
- 聊天问答体验突出
- 引用来源表达可信
